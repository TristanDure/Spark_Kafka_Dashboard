## 一、环境相关

### 1. 启动命令
```bash
cd E:\code\周记\大数据大作业\Spark_Kafka_Dashboard

# 完整模式（推荐，含PySpark）
python run.py --spark

# 轻量Demo模式（零依赖，纯Python）
python python_dashboard/app_demo.py
```

### 2. 启动前必须杀干净残留进程
```bash
# Windows上可能有残留python进程占用5000端口
netstat -ano | findstr 5000   # 找到PID
taskkill /F /PID <PID>        # 杀掉
```

### 3. PySpark注意事项
- **Java版本**：当前Java 23可运行，但必须用 `local[1]` 单worker模式（已配置）
- **环境变量**：启动前确认 `PYSPARK_PYTHON=python`（run.py 已自动设置）
- **Spark UI端口**：默认4040，如果被占用会自动用4041
- **HADOOP_HOME警告**：Windows上缺少winutils.exe，不影响功能，Spark会自动降级

### 4. Kafka说明
- Kafka 3.6.0 已下载到 `E:\code\周记\大数据大作业\kafka_2.13-3.6.0/`
- Docker方案因网络问题暂不可用，Kafka直接跑需要Java 8/11
- **当前系统不需要Kafka运行**——代码设计了fallback机制，无Kafka时自动使用本地管道

## 二、运行模式说明

| 模式 | 命令 | 依赖 | 适用场景 |
|------|------|------|----------|
| Demo | `python app_demo.py` | 仅Flask | 快速演示可视化效果 |
| PySpark | `python run.py --spark` | +PySpark | 完整大数据处理演示 |
| 全组件 | 需Docker | +Kafka+HDFS | 生产环境或答辩展示 |

### 三种模式对比

```
Demo模式：
  数据生成 → 直接计算 → ECharts
  组件覆盖：0个大数据组件（纯演示）

PySpark模式（当前推荐）：
  数据生成 → PySpark(SQL+Streaming) → HDFS(模拟) → ECharts
  组件覆盖：Spark Core + Spark SQL + Spark Streaming + HDFS = 4个

全组件模式（Docker可用时）：
  数据生成 → Kafka → PySpark → HDFS → ECharts
  组件覆盖：Kafka + Spark + HDFS = 5个
```

## 三、答辩注意事项

### 1. 演示前准备
- [ ] 确认5000端口未被占用
- [ ] 启动后打开浏览器，测试所有7个页面都能加载
- [ ] 确认数据实时刷新（每2秒一批，折线图应持续滚动）
- [ ] **网线拔掉/断网不影响**——ECharts从bootcdn加载，首次需缓存

### 2. 如果现场演示PySpark模式
- 启动后第一分钟Spark初始化较慢，属正常现象
- 控制台会输出WARN日志，这是Spark的正常行为，**不是错误**
- 如果看到 `Python worker failed to connect back`，重启即可

### 3. 答辩话术关键点
- 强调 **5个大数据组件** 的协同
- 强调 **实时+离线双模式**
- 强调 **完整数据链路**
- 提到 Windows环境上的适配工作（体现了解决实际问题的能力）

## 四、已知问题 & 解决方案

| 问题 | 影响 | 解决 |
|------|------|------|
| `socket.io` 400错误 | 页面刷新时偶尔出现 | 刷新页面即可，不影响数据流 |
| PySpark worker超时 | Windows多worker不兼容 | 已设定 `local[1]` 单worker |
| Docker镜像拉取超时 | Kafka/Docker不可用 | 已设计fallback机制，不影响演示 |
| winutils.exe缺失警告 | 仅控制台提示 | 不影响功能，可忽略 |
| ECharts CDN加载 | 首次需联网 | 已缓存在浏览器，断网后仍可用 |
